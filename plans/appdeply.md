# VillKro — Complete Production Flow Plan: PC to Play Store

This is the **single source of truth** for shipping `apps/customer-app` to the Google Play Store. It is tuned to **this exact repo** (Expo SDK 54, React Native 0.81, EAS already configured, monorepo with `apps/{api,web,admin,customer-app}`) and to **building on your own Linux PC** — zero EAS Cloud spend.

Read it **top to bottom once** before doing anything. The plan is deliberately click-by-click so a first-timer can follow it without guessing.

---

## 0. TL;DR (the one paragraph you actually need)

You work on a `dev` branch and push as much as you want. When you merge to `main`, **your Linux PC** (a self-hosted GitHub Actions runner on the same machine) wakes up, builds the Android `.aab` for free using EAS Local, uploads it to the Play Console, and ships an EAS Update (OTA) to every installed phone — all in one shot. JS-only changes go to users in seconds via OTA. Native changes go through the usual 1–2 day Play Store review.

**Two decisions baked in (per your instruction):**
- **Builds run on your Linux PC (`eas build --platform android --local`).** No EAS Cloud, no queue, no fees.
- **Every merge to `main` auto-publishes an EAS Update** on the production channel. Native changes still need a Play Store review; JS changes are instant.

## 1. How the Play Store Flow Actually Works

- **First Time Upload (new app)**: 3–7 days for Google review.
- **Subsequent Full Updates** (new `.aab` with native changes): 1–2 days review.
- **JS-only changes** (text, colors, layout, business logic): **no Play Store review** — delivered instantly via EAS Update. Google explicitly allows this as long as you don't add new native hardware features.

> Rule of thumb: if the change touches only files under `apps/customer-app/src/**` and does **not** add a new native package (e.g. camera, bluetooth, maps SDK), OTA covers it. If you `npx expo install <some-native-package>`, you must do a **full `.aab` build** to the Play Store.

## 2. Auto-Update Policy (everything you push to `main` reaches users)

> **Warning — read once:** every merge to `main` will produce an EAS Update and an `.aab` upload. There is **no preview channel and no manual gate**. If you merge broken code to `main`, broken code reaches users in seconds.

Mitigations we are still adding:
- **`dev` is the safe branch.** Day-to-day work happens there. Only merge to `main` when you're willing to ship.
- **CI runs first.** If `npm run lint` or `npm test` fails, the workflow stops — no build, no upload, no OTA.
- **Bump policy below** controls whether a merge triggers a full Play Store upload (major) or just an OTA (minor).

| Bump you choose in commit / PR | What happens on merge to `main` | Play Store review? |
|---|---|---|
| `feat:` / `BREAKING:` / native package added | Full `.aab` rebuild + Play Store upload + OTA | Yes (1–2 days) |
| `fix:` / `chore:` / `style:` / `refactor:` / `docs:` / no native | OTA only (no Play Store upload) | No — instant |

How it's enforced automatically: the workflow detects if a commit in the merge added/changed/removed any entry in `apps/customer-app/package.json` `dependencies`/`devDependencies`/added native modules. If yes → major/full build. If no → minor/OTA only.

## 3. What is EAS Local

**EAS** compiles your JavaScript into a native Android `.aab`. **EAS Local** runs that compile on **your own machine** using your own CPU. There is no queue, no cloud upload, and no per-build fee.

Requirements on your Linux PC:
- **JDK 17** (Temurin recommended)
- **Android SDK Platform 35** + **Build-Tools 35.0.0** + **Platform-Tools** + **NDK 26.1.10909125** (RN 0.81 needs this NDK)
- ~10 GB free disk for SDK + caches
- Your PC must be **on and connected to the internet** when you push to `main`

We will install all of this in Step 0 below.

## 4. Files & Folders `eas build --local` Will Create (and how we ignore them)

`eas build --local` runs `expo prebuild` internally, which generates a real native Android project from your `app.json`. **This creates an `android/` folder with thousands of generated files** (~3,000 files, ~200 MB). Plus Gradle caches and build outputs.

**None of these should ever be committed.** They are reproducible from your `app.json` + `package.json` on every build, so committing them only causes merge conflicts.

The root `.gitignore` (already updated) excludes:
- `apps/customer-app/android/` — generated by `expo prebuild`
- `apps/customer-app/android/.gradle/` — Gradle cache
- `apps/customer-app/android/build/` and `app/build/` — build outputs
- `apps/customer-app/android/local.properties` — points to your local SDK, machine-specific
- `apps/customer-app/android/.kotlin/` — Kotlin cache
- `**/.cxx/` — native (C++) build cache
- `*.aab`, `*.apk`, `*.jks`, `*.keystore` — built artifacts and signing material
- `play-store-key.json`, `**/*-service-account.json` — Play Store credentials
- `.actions-runner/` — the GitHub self-hosted runner install dir (we put it *outside* the repo to be safe, but this is a belt-and-suspenders)
- `android-sdk/` — in case you ever install the SDK inside the repo

If you ever see a `git status` showing files inside `apps/customer-app/android/`, stop and check `.gitignore` — those should never appear.

> **Heads-up:** because `android/` is generated, do not edit files inside it for the release build. Anything custom (e.g. `MainApplication.kt`) must go in an Expo config plugin. We'll keep the project 100% standard for now.

Pushes to `apps/api/`, `apps/web/`, `apps/admin/`, root configs, or docs are ignored by the Play Store workflow (your existing `ci-api`, `ci-web`, `ci-admin`, and `deploy.yml` will still run for those).

## 6. The Final Workflow (Code Daily, Merge to Ship)

**Branch layout**
- `main` — protected. Merging here triggers the full pipeline (build + submit + OTA).
- `dev` — your day-to-day branch. Push freely; nothing reaches the Play Store.
- `feature/*` — optional short-lived branches off `dev`.

**Step A — Daily coding (safe)**
1. `git checkout dev`
2. Edit, commit, `git push origin dev` as many times as you want.
3. The Play Store workflow ignores this branch.

**Step B — Trigger (the "one thing")**
When you are ready to ship:
```bash
git checkout main
git pull origin main
git merge dev
git push origin main
```
Or click **"Merge Pull Request"** on GitHub from `dev` → `main`.

**Step C — What happens automatically on your PC**
1. GitHub pings the **self-hosted runner** that lives on your Linux PC.
2. The runner runs `npm ci`, lint, tests in `apps/customer-app` — must pass.
3. Bump-detection step: did this commit change a native dependency?
   - **No** → run `eas update --branch production --non-interactive --message "<commit msg>"`. Done in ~30 seconds. Users get it on next app open.
   - **Yes** → run `eas build --platform android --local --non-interactive --profile production` (~10–20 min on your CPU). This creates `apps/customer-app/android/` (auto-ignored), produces a signed `.aab`, then `eas submit --platform android --latest --non-interactive` uploads it to Google Play (Internal testing track). Then `eas update --branch production` ships the JS too.
4. The runner reports success/failure back to GitHub.

You can close the lid; the runner logs are visible in the GitHub Actions tab.

## 7. One-Time Setup (do these once, in this order)

### Step 0 — Install the Android toolchain on your Linux PC (15–25 min, one time)

Your PC is missing Java and the Android SDK. Run these in order. All installs go to `/opt/android-sdk` (system-wide) — **not** inside the repo, so nothing leaks into Git.

```bash
# 0a. JDK 17
sudo apt update
sudo apt install -y openjdk-17-jdk-headless unzip
java -version   # must show 17.x

# 0b. Android command-line tools → /opt/android-sdk/cmdline-tools/latest
sudo mkdir -p /opt/android-sdk/cmdline-tools
cd /tmp
curl -fsSL -o cmdline.zip https://dl.google.com/android/repository/commandlinetools-linux-11076708_latest.zip
sudo unzip -q cmdline.zip -d /opt/android-sdk/cmdline-tools
sudo mv /opt/android-sdk/cmdline-tools/cmdline-tools /opt/android-sdk/cmdline-tools/latest

# 0c. Persist env vars in your shell
cat >> ~/.bashrc <<'ENV'
export ANDROID_HOME=/opt/android-sdk
export ANDROID_SDK_ROOT=/opt/android-sdk
export JAVA_HOME=$(dirname $(dirname $(readlink -f $(which java))))
export PATH=$PATH:$ANDROID_HOME/cmdline-tools/latest/bin:$ANDROID_HOME/platform-tools:$ANDROID_HOME/emulator
ENV
source ~/.bashrc

# 0d. Accept licenses + install SDK 35, build-tools 35.0.0, platform-tools, NDK 26.1.10909125
yes | sdkmanager --licenses
sdkmanager "platform-tools" "platforms;android-35" "build-tools;35.0.0" "ndk;26.1.10909125"
```

Verify:
```bash
java -version                          # 17.x
echo "JAVA_HOME=$JAVA_HOME"
echo "ANDROID_HOME=$ANDROID_HOME"
sdkmanager --list_installed             # must list android-35, build-tools;35.0.0, ndk;26.1.10909125, platform-tools
adb --version
df -h /opt | tail -1                    # should show ~5-10 GB used
```

> **Why these exact versions:** your `package.json` pins `react-native@0.81.5` which requires NDK `26.1.10909125`; `app.json` sets `targetSdkVersion: 35`, so SDK 35 is mandatory; JDK 17 is what AGP 8.x and Gradle 8.x want.

### Step 1 — Install the self-hosted GitHub Actions runner on your PC (10 min, one time)
This is how GitHub tells **your** PC to start a build. We install it **outside the repo** to keep build state out of Git.

1. GitHub → your repo → **Settings → Actions → Runners → New self-hosted runner**.
2. Choose **Linux, x64**. GitHub shows you 4 copy-paste commands.
3. Open a terminal and run them **inside a folder outside the repo**:
   ```bash
   mkdir -p ~/actions-runner-villkro && cd ~/actions-runner-villkro
   # (paste GitHub's curl to download the tarball)
   # (paste tar xzf)
   ./config.sh --url https://github.com/<your-username>/<your-repo> --token <TOKEN> --labels villkro-android,self-hosted,linux,x64
   ```
4. Install as a systemd service so it auto-starts on boot and survives reboots:
   ```bash
   sudo ./svc.sh install
   sudo ./svc.sh start
   ```
5. **CRITICAL: env vars for the runner service.** systemd services don't load `~/.bashrc`, so Java/Android won't be found unless we inject them. Run:
   ```bash
   sudo systemctl edit actions.runner.<your-org>-<your-repo>.villkro-android.service
   ```
   Paste (adjust `JAVA_HOME` if yours differs):
   ```ini
   [Service]
   Environment="JAVA_HOME=/usr/lib/jvm/java-17-openjdk-amd64"
   Environment="ANDROID_HOME=/opt/android-sdk"
   Environment="ANDROID_SDK_ROOT=/opt/android-sdk"
   Environment="PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin:/opt/android-sdk/platform-tools:/opt/android-sdk/cmdline-tools/latest/bin"
   ```
   Then:
   ```bash
   sudo systemctl restart actions.runner.<your-org>-<your-repo>.villkro-android.service
   ```
6. Back in GitHub → Runners, the dot should be **green (idle)**. Click on it to confirm the label is `villkro-android`.

> Keep your PC on while you expect a build. Sleep/hibernate will pause the runner. Locking the screen is fine.

## 12. What Can Go Wrong (and the fix)

| Symptom | Cause | Fix |
|---|---|---|
| Runner dot is red in GitHub → Runners | `svc.sh` stopped | `sudo ./svc-runner.sh start` (or `sudo systemctl start actions.runner.*`) |
| `eas build --local` fails: "JAVA_HOME not set" | Step 1 env vars not injected into runner service | Re-run `sudo systemctl edit …` block from §7 Step 1 and `systemctl restart` |
| `eas build --local` fails: "SDK location not found" | `ANDROID_HOME` not set for the runner service | Same fix as above |
| `apps/customer-app/android/` shows up in `git status` | `.gitignore` entry missing or path typo | Re-check §4 list, fix `.gitignore`, run `git status --ignored` |
| Build stuck at "compiling native code" | NDK missing or wrong version | `sdkmanager "ndk;26.1.10909125"` |
| "Build failed: keystore not found" | Skipped §7 Step 6 | Run `npx eas credentials` and generate keystore |
| "Submit failed: 403" | Service account lacks track permission | Play Console → Setup → API access → grant "Release to track" |
| "Update not appearing on device" | App was built without `expo.updates` config | We add it in §8 step 1 |
| "Two different signing keys" warning | You uploaded an `.aab` you built locally with a different keystore | Always let EAS manage the keystore; never build with `expo run:android` for releases |
| Review rejected for "test account required" | You didn't provide login credentials in §7 Step 3 App access | Add a real test account to the App access section |
| `adb` not found in workflow logs | `platform-tools` not installed | `sdkmanager "platform-tools"` |
